# Doc-Writer Instructions

Generate comprehensive documentation for a scaffold MCP tool app.

## Documents to Generate

### README.md
- App name and one-line description
- What problem it solves
- Tool list with descriptions
- MCP server URL
- Quick installation: clone, configure, connect

### docs/discovery-story.md
- Write as a brief narrative, not a data dump
- How the idea was discovered (source, community, signal)
- The "aha" moment: what made this idea stand out
- 2-3 paragraphs, engaging tone

### docs/design-decisions.md
- Key architectural choices and WHY (not just WHAT)
- Data model decisions
- Tool design trade-offs
- What was considered but rejected

### docs/challenges.md
- What went wrong during build/test
- How issues were fixed
- Iterations needed
- Lessons learned

### docs/testing-summary.md
- Which personas tested
- What each found
- Pass/fail breakdown
- Key findings and fixes

### docs/quick-start.md
- 3 steps max
- Must be copy-pasteable in under 2 minutes
- Include exact commands

### docs/detailed-guide.md
- Full usage guide
- Example for every tool
- Common workflows
- Tips and tricks

### promo/reddit-post.md
- Title for r/sideproject
- Body written as "I built this because..."
- Genuine and helpful, not salesy
- Include what it does, how to try it, what's next

### promo/twitter-thread.md
- 3-5 tweets
- First tweet is the hook
- Include a tool example
- End with link to try it

### promo/hn-post.md
- Show HN title
- Brief description
- Technical details that HN would appreciate

## Tone Guidelines
- Discovery story: narrative, engaging
- Design decisions: technical but accessible
- Quick start: minimal, action-oriented
- Promo: genuine, builder-community voice
- All docs include scaffold framework attribution
